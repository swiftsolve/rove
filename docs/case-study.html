Case Study · Product Engineering

A network monitor that tells the truth right now.

Rove shows the honest, live state of your network — and stays correct the instant anything changes.

Live Role · Solo — product, design, engineering Stack · Rust · Tauri v2 · React ~21k LOC

The bet

Network state should never feel stale.

A modern connection isn't one fixed thing. You pull a cable, join a new SSID, drift between bands, walk from one router to the next. Most tools treat the network as static and poll on a slow timer — so they're routinely wrong right when you open them to find out why something changed.

Rove is built around that motion. The whole product rests on one wager: a monitor that's ever wrong isn't a monitor. Being currently true is the entire job — and the name, Rove, names the behavior.

Judgment under tradeoffs

Three decisions worth showing.

DECISION 01

Make “current” the product, not a feature

Instead of polling every 15 seconds, the backend watches the OS routing table and re-reads the source of truth the moment anything changes, nudging the UI within about a second. Slower platforms fall back to the poll and still work.

The betThe extra plumbing — a push channel alongside request/response — exists entirely to protect one feeling: that what you're looking at is true.
DECISION 02

Answer a hard question with cheap signals — honestly

“What's on my LAN?” has no clean OS API. Rove fuses seven unprivileged signals — ICMP sweep, TCP probe, mDNS, SSDP/UPnP, reverse DNS, an HTTP banner grab, and NetBIOS — and reconciles them with a trust-weighted vote to name and classify each device in a couple of seconds, no admin rights.

The tradeoff I ownedA device that blocks ping and announces nothing can still hide. Rather than fake exhaustiveness, the UI defers to the router as authoritative. Naming the limit is better product than hiding it.
DECISION 03

Degrade gracefully instead of erroring

Every platform exposes this data through different tools, any of which can be missing. The service layer treats a missing tool as “no data,” not a failure — the next probe is tried, and missing values render as a quiet “—”, never a crash.

Why it mattersUtilities get judged in their worst moment — the weird laptop, the locked-down OS. Graceful degradation is what makes it trustworthy there.

The last 10%

The polish that makes a utility feel like a product.

  • Speed-test Stop settles the UI immediately, instead of waiting for the backend to unwind.
  • Services show a clear “Down” state rather than an ambiguous blank.
  • Frameless, minimal chrome with a staggered entrance — quick and quiet, not chatty.
  • Copy tuned deliberately — subtitles trimmed back to concise one-liners.
None of these were in a spec. They're the difference between “it works” and “someone cared.”

Constraints

Small on purpose, and it stays honest under pressure.

  • Solo, cross-platform. Platform differences are runtime checks, so every code path typechecks on every OS and CI builds all three bundles.
  • Lightweight by design. A ~5 MB installer; a memory-safe Rust backend with no Node runtime in the UI process — the webview can only call the specific commands I expose.
  • Known cost, stated plainly. A speed test at multi-gigabit rates moves 1–2 GB — inherent to measuring throughput. Rather than hide it, the app warns you. Trust over polish.

Next

Where it goes from here.

  • Turn the router-authoritative caveat into a feature — optional import from the router's device list to close the hidden-device gap.
  • History and alerting on metrics that already stream — throughput, gateway loss. The pipeline is already there.
  • A packaged auto-update channel, so “small and fast” survives distribution.

Takeaway

One clear bet, carried all the way through.

Rove is a small product with a single thesis — be true right now — pushed through the architecture, the tradeoffs, and the last-10% polish. The case where I set the direction and built the systems to honor it.
